The syntax to using scp is just like how you would use cp, except you can add additional information about which computer you are trying to copy from/to, and which account within those computers.

The following is the syntax for copying a file from the local machine to a remote computer:

$ scp /some/local/file.txt username@remotehost.edu:/some/remote/directory

Goal: Learn about access permissions to a file, and how to modify permissions.

The chmod command stands for change mode. It allows you to change the access permissions to a file, i.e. who gets to read, write, and execute certain files or directories.

$ chmod [options] mode[,mode] file1 [file2 ...]

Let’s start by understanding the permissions of files. List all the files in your home directory in long format to see their permissions by typing: 

$ ls -l ~

For each file, you should see something like:

-rw-r----  11 cs15wxx cs15w  4096 Jan 01 00:00 filename

File permissions control how others on the system can interact with your files and directories. There are three basic permissions: read (r), write (w) and execute (x). Users of the system are classified into one of three types: user (aka owner, the one who created the file), group (the group assigned to the file) and others (aka world, anyone on the system). There are permissions associated with each type. The permissions are displayed as a string where each character in the string refers to a specific permission. The first character is the file type, the next three characters are the read/write/execute permissions for the user (owner), the next three are the r/w/x for group and the final three are the r/w/x for others (aka world).

The string -rw-r----- is the permissions on the file. The first character is - which means this is a regular file (you'll see "d" here for a directory for example). The user (owner) permissions are rw- which means there is read and write permissions but not execute. The group permissions are r-- which means they only have read permissions. The other (world) permissions are --- which means they can’t read, write, or execute. These are the default permissions for a file on ieng6. The owner of this file is cs15wxx and the group is cs15w.

There are two ways to specify permissions to chmod: using octal numbers and using symbolic names. For octal numbers, view the set rwx as the 3 digit binary number 111. A 1 means the permission is set and a 0 means the permission is not set. So, binary 100 would be read only and it is equivalent to the octal number 4. So the number 754 would set owner permission to 7 (rwx), the group permission to 5 (r-x) and the world permission to 4 (r--). For more information, you can look at the wikipedia entry for examples: http://en.wikipedia.org/wiki/Chmod#Octal_modes.

Goal: Search for files from the command line

The find command searches by file names and path. The syntax is:

$ find [where-to-look] [criteria]

Let’s start by searching for the squirtle file.

$ cd
$ find . -iname “squirtle”

The [where to look] in this case is . which specifies the current directory. The [criteria] is -iname squirtle which means searching by file name, ignoring the case, that matches squirtle. We can also use wildcards and search for files with a given extension.

$ find . -iname “*.txt”

The gzip command compresses files to save space. It stands for GNU zip. Type the following to compress the practice.txt file. If you are running into any issues with

It will compress it and add a gz extension. To uncompress the file, use GNU unzip:

Let’s undo that by recursively unzipping them.

$ gunzip -r pokemon

This happens because gzip only compresses individual files, so you can’t zip up a folder like you can on Windows. Therefore the gzip command is often used along with the tar command. We have to first create a tarball of the directory, and then we can compress that tarball.

$ cd ~/lab3B
$ tar cvf pokemon.tar pokemon
$ gzip pokemon.tar
$ ls

This will create a pokemon.tar.gz file. To extract the files:

$ gunzip pokemon.tar.gz
$ tar xvf pokemon.tar

Goal: Examine and analyze the Exit Codes of UNIX programs

Exit Codes refer to the values unix programs return. These Exit Codes follow a similar structure and pattern. They can be useful for diagnostic output, as well as making dynamic decisions in a shell script. To get started lets write a quick shell script called exitstatus.sh inside of ~/lab3A.

#!/bin/bash

echo -e "Successful execution: echo \"hello world\""
echo -e "===================="
echo "hello world"
#Exit status returns 0 because the above command was successful
echo -e "Exit Status" $?

echo -e "\nIncorrect usage: ls --option"
echo -e "====================="
ls --option
#Incorrect usage, so exit status will be 2.
echo "Exit status" $? 

echo -e "\nCommand Not found: bashscript"
echo -e "====================="
bashscript
#Exit status returns 127, because bashscript command not found
echo "Exit status" $? 

echo -e "\nCommand is not an executable: ./lab3_test.txt"
echo -e "============================="
touch lab3_test.txt
./lab3_test.txt
#Exit status returns 126, because its not an executable.
echo "Exit status" $?

Before running the script, be sure to use chmod to make the script executable.

The first thing to notice is the $? - this special variable holds on to the exit code from the previously executed function. By printing this after the comand, we can see the output of the command and then match it with the exit code returned. In general, there is a pattern to the exit codes.

Exit Value Exit Status
Zero Success
Non-Zero Failure
2 Incorrect Usage
127 Command Not Found

Goal: Use the less command to make large blocks of text on the console easier to read.

As the goal says, the less command makes large blocks of text on the console easier to read. less is a better version of the more command, which does essentially the same thing, except less has more features. less is more, as one might say. Commands that break large bodies of text into smaller pages are called pagers. Pagers are often invoked by other external commands to facilitate the paging of their output, a good example of this is the man command. When issuing the man command it filters its output through the less command by default. So everything that worked while using man will also work while using less. Remember ps -ef had output that was a lot of text? We can view it easier by using less. 

$ ps -ef | less

While viewing in less, you can use the following commands:

Command Description
/apple or ?apple Forward and backwards searches just like in vim.
b Goes up/back a page, in vim it’s ctrl-b. pgup does the same thing.
f or space Goes forward/down a page. In vim it’s ctrl-f. pgdown does the same thing.
q quit

Section C: Scientific Method of Debugging

In this section, we will be introducing the Scientific Method of debugging. The Scientific Method involves the following steps:

  1. Identify the behavior you want to explain.
  2. Form a hypothesis that might explain that behavior.
  3. Conduct an experiment that tests the hypothesis.
  4. If the experiment contradicts the hypothesis, revise it or replace it with a new hypothesis and go back to step 3.

When applied to the debugging process, it might be more along these lines:

  1. Identify the bug you want to fix.
  2. Make a guess as to what the program is doing to cause the bug.
  3. Conduct a test to see if the guess is correct.
  4. If the test contradicts the guess then revise it or replace it with a new guess and go back to step 3

These steps can be formalized and broken down further into distinct parts that make the write up for debugging using this method. We usually like to think of these as the following:

  • Symptom* (a discrepancy between desired and actual behavior)
  • Hypothesis (a potential cause of the symptom)
  • Experiment (a manipulation of the artifact)
  • Prediction (the expected outcome of the experiment)
  • Observation (the actual outcome)
  • Conclusion (the hypothesis is falsified, or supported)

Note: If you have multiple programs running you should stop each instance by clicking Terminate on the Console and Debug perspectives.   1. Step Execution: Do not guess how your program is executing. Instead, watch the debugger execute your program.

  • Step into executes the next line in your program. If the current line contains a method, it will stop on the first line of the called method.

  • Step over executes the next line of your program while skipping over any method calls. This is equivalent to continuing execution to the next line in the current file.

  • Step return continues execution until the program finishes the current method.

  • Resume continues execution as normal.

    The debugger shows the values of the local variables in their current state, in the top right pane. The values get updated as you step through the program. The top right pane also lists all active breakpoints.

  1. Watch expressions allows you to see the results of evaluating expressions whenever the program is paused. After stopping the execution of your program, highlight an expression, right click and select watch. The expression should now be in the top right pane. Eclipse will show you the evaluated value. Be careful with expressions that create side effects, because Eclipse actually executes the highlighted code each time, to show its result.

Example: Suppose that you are curious about the logic in an if condition, or any other expression like this.

if(!isCondition1True() && isCondition2True())

In this part of the lab, you will implement a Insertion Sort method for the LinkedList class. The algorithm consists of inserting one element at a time into the previously sorted part of the array, moving higher ranked elements up as necessary. To start off, the first (or smallest, or any arbitrary) element of the unsorted array is considered to be the sorted part. The algorithm is illustrated below:

The following website provides an animation that demonstrates the steps of Insertion sort and also contains the pseudo code: http://www.ee.ryerson.ca/~courses/coe428/sorting/insertionsort.html

Pseudo Code for Insertion Sort:

From Wikipedia: Pseudocode is an informal high-level description of the operating principle of a computer program or other algorithm. It uses the structural conventions of a programming language, but is intended for human reading rather than machine reading.

Note: This is pseudo code, not working code!

for j <- A[1] to length[A] do

    //Insert A[j] into the sorted sequence A[1 . . j - 1].
    key <- A[j]
    i <- j - 1

    while i > 0 and A[i] > key do

        A[i + 1] <- A[i]
        i <- i - 1

    A[i + 1] <- key

<- is an assignment operator, equivalent to =

do represents the block of code to be done after the previous statement. Pay attention to the indentation!
