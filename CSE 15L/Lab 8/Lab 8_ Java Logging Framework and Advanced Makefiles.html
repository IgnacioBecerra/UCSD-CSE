Goal: Properly Configure the Java Logging Framework

Open the file logging.properties in a text editor of your choice.

The first thing we need to do is to specify the Handlers that we will be using with our Logger objects. Jump to section 1, titled "Global logging properties", and uncomment the following line (comments start with a # and go to the end of the line):

#handlers=

Then add two handlers: 

handlers=java.util.logging.FileHandler, \
     java.util.logging.ConsoleHandler

Each log message has an associated log Level. The Level specifies the importance and urgency of a log message. Levels are really integers, with higher values indicating higher priorities.

The Level class defines seven standard log levels as named constants, ranging from FINEST (the lowest priority, with the lowest value) to SEVERE (the highest priority, with the highest value).

Question 1: Write out the seven standard log levels from FINEST to SEVERE.

Now, we want to set the default global logging Level. We will set the global level to FINEST. This level will apply to all Loggers and Handlers, unless specifically overridden, and means that no messages will be ignored. To change this, we can then specify another level for a particular Logger or Handler. This will allow us to efficiently control which messages we want to print to the console or to the file.

In logging.properties go to section 2, "Default global logging level", and uncomment the following line:

# .level=

Change it to read:

.level=FINEST

Next, we will set the level of the Handlers separately. Jump to section 3, "Console Handler", and uncomment the following line:

# java.util.logging.ConsoleHandler.level=

Change it to read:

java.util.logging.ConsoleHandler.level=FINEST

so that the level is set to FINEST for now. (This is the same as the global level you set above; we will be able to change it later though, which will be useful for controlling tracing output.)

Do the same for the FileHandler; set its level to FINEST, at least for now.

Look at the "pattern" property (but do not change it) in section 5. It specifies the pattern of the name of the logfile that FileHandler should use.log for your program. Here it is specified as uniq.log.

Finally, jump to section 7 and set the output formatter of both of the Handlers to SimpleFormatter. For example:

java.util.logging.FileHandler.formatter=java.util.logging.SimpleFormatter

Your properties file is now ready. We will use it to configure our Loggers for debug tracing. Our Java application can now be launched and pointed to the configuration file by passing the java.util.logging.config.file command line parameter as illustrated here:

java -Djava.util.logging.config.file=

Goal: Understanding how to use Debug Tracing in the Java Logging API

Tracing involves logging messages which report the state of the application at different stages of execution. In the Java logging framework, there are three levels that should be used for tracing: FINE, FINER and FINEST.

FINE: Use this level for significant events that explain the flow or state of the system when trying to understand or debug a problem. These are usually at the level of object creation, catch clauses for exceptions that do not constitute errors, and so on.

FINER: There is a set of Logger methods that generate messages with a FINER level. These are entry(...), exit(..), finer(..) (This is a method named finer; different from a logging level called FINER) and throwing(..). Entering and exiting should be used for tracing method entry and exit; arguments and return values can be specified. Finer can be used for any tracing that is more verbose than desired for fine, but is not method entry/exit or used before throwing an exception. Throwing should be used when you are about to throw or re-throw an exception.

FINEST: Finest is usually thought of as developer or debug tracing. This is used during development, when attempting to learn the behavior of a system at a fine level of detail and when trying to diagnose difficult problems once the code is released. This level of tracing can be used to track the state of instance variables in a class e.g. before a method call, inside a method or a loop and after a method call.

You are required to implement tracing at the FINER and FINEST levels for this lab. The file Lab8.java contains a skeleton that a developer might use to trace the execution of a program in order to uncover the bugs in the removeDuplicates() method.

It is a simplified version that sorts the contents of a file and does a case insensitive comparison of consecutive lines. It outputs a file with consecutive identical lines collapsed to just one copy.

The version of the program given to you in the Lab8.java file has one bug. It is your job to make it meet the specification above and work correctly on the sample input. You will use the logger output to find and fix the bug.

Open Lab8.java and get started on the debugging exercise. The Logger has already been defined for you at the bottom of the class:

private Logger logger = Logger.getLogger(Lab8.class.getName());

At the top of the run(..) method, type in the following code, to implement FINER tracing.

logger.entering(Lab8.class.getName(), "run(String[] args)", args);

Do the same at the end of the run method:

logger.exiting(Lab8.class.getName(), "run(String[] args)");

We create a message at level Fine to indicate whether the private member linked list is empty, as an example:

//FINE Log : Whether the linked list is empty or not? 
logger.fine("words.empty() = " + words.isEmpty());

Now look for the comments in function run() which indicate that FINE log should be added: //FINE Log : What is the size of the linked list?

factorial.cpp       sum_4_numbers.cpp       summation.cpp

For this subtask ("Archiving subtask"), you must write a Makefile in the cpp/lib directory to do the tasks below. To create a Makefile, open up a text editor of your choice and save the file as "Makefile":

default target:

Compile all source files into object code files.\ Using the archiver, ar, archive the object code files into an archive named libFunctions.a. You can use the command below for this step:

ar rcs libFunctions.a factorial.o sum_4_numbers.o summation.o

Useful: http://www.ibm.com/support/knowledgecenter/ssw_aix_61/com.ibm.aix.cmds1/ar.htm

Example for compiling source file, file_name.cpp into object code file: 

g++ -c file_name.cpp

clean target:

  1. Delete the archive "libFunctions.a".
  2. Delete the object code files in this directory. \ Use rm -f to remove specified files.

new target:

  1. "make" the "clean" target.
  2. "make" the "default" target.

C++ subtask

For this subtask ("C++ subtask"), you must write a Makefile (in the cpp directory) to do the following:

default target:

  1. "make" the "lib/" directory contents. To execute Makefile of subfolder lib, from inside parent directory Makefile, use:

Replace target with new, clean, etc. as required.

  1. Compile the source file in this directory (cpp), "main.cpp", into an object code file.
  2. Link all object code files into a executable named "cpp.exe". Remember that the "Archiving subtask" creates an archive (.a file) on which "main.cpp" depends. Link that archive file as well.
  3. Copy "cpp.exe" into the directory "../bin/".

Hints:

  • Example for compiling file_name.cpp:

  • Example for linking files: file_name1.o, file_name2.o, archive_file.a into file_name.exe:

clean target:

  1. Clean the "lib" directory.
  2. Delete the object code files in this directory.

new target:

  1. "make" the "clean" target.
  2. "make" the "default" target.

Goal: Consolidate subtasks into one Makefile task

This is the rest of "Makefile task", which has you producing yet another Makefile to tie all the subtasks together. Make sure to use .PHONY when creating the Makefile (more info about that here). This Makefile in makefile_assignment directory must be capable of carrying out the following:

.PHONY: cpp java

default target:

  • "make" the targets "cpp", and "java"

  • "make" the contents of the directory makefile_assignment/cpp/

java target:

  • "make" the contents of the directory makefile_assignment/java/

  1. Clean the "cpp/" directory.
  2. Clean the "java/" directory.
  3. Delete all executables in the "bin/" directory.

new target:

  1. "make" the "clean" target
  2. "make" the "default" target
